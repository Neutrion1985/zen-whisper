# Правила ведения CHANGELOG.md

## 📋 Формат файла

CHANGELOG.md — это единый двуязычный файл с историей изменений. Расположение: **корень проекта** (`/CHANGELOG.md`).

---

## 🌐 Двуязычная структура (Interleaved + Bullets)

Каждая версия и каждый пункт изменений должны содержать текст на двух языках одновременно. Используются префиксы **- ENG -** и **- РУС -**.

### Пример структуры:

```markdown
# Changelog / Журнал изменений
All notable changes to ZenWhisper will be documented in this file.
Все значимые изменения ZenWhisper документируются в этом файле.

---

## [X.Y.Z] — YYYY-MM-DD
### Added / Добавлено
- ENG - **Component Name**: English description of the feature.
- РУС - **Имя компонента**: Описание функциональности на русском.

- ENG - **Next Feature**: Another technical update.
- РУС - **Следующая фича**: Еще одно техническое обновление.
```

---

## 📝 Категории (Keep a Changelog)

Используйте двойные заголовки категорий:
1. **### Added / Добавлено** — новая функциональность
2. **### Changed / Изменено** — изменения в существующей функциональности
3. **### Improved / Улучшено** — технические улучшения
4. **### Fixed / Исправлено** — исправления ошибок
5. **### Removed / Удалено** — удалённая функциональность

---

## ✅ Правила ведения

1. **Новая версия — всегда сверху**.
2. **Маркированные списки обязательны**: Каждая строка должна начинаться с дефиса (`- `).
3. **Префиксы**: Используйте `- ENG -` и `- РУС -`.
4. **Разделение парами**: Между каждой парой (ENG + РУС) должна быть **пустая строка** для визуального разделения.
5. **Жирный шрифт**: Название компонента или функции должно быть выделено **жирным** на обоих языках.
6. **Дата**: Формат `YYYY-MM-DD`.

---

## 📦 Связь с GitHub Releases

При создании Release на GitHub:
- Копируйте только английскую часть (`- ENG -` секции) или двуязычную в новом формате.
- Теги в Git должны совпадать с версиями в CHANGELOG.
